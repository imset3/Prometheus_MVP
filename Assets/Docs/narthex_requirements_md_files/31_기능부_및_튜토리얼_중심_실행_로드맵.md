# 31. 기능부 및 튜토리얼 중심 실행 로드맵

## 1. 이번 로드맵의 목표

이번 단계의 목표는 CHAPTER_01 전체 완성이 아니다.

다음 한 줄을 실제 Unity에서 끝까지 실행하는 것이 목표다.

```txt
Boot
→ Tutorial 시작
→ 이동/점프/대시/공격 학습
→ 모듈 장착 및 사용
→ 송신탑 활성화
→ 헬테 전투
→ 헬테 처치
→ 보스 모듈 트리 영구 해금
→ 튜토리얼 완료
→ CHAPTER_01 진입
```

이 흐름을 위해 필요한 시스템만 먼저 구현한다. CHAPTER_01의 모든 스테이지, 최종 보스, 완성 아트, 고급 연출은 이번 목표에서 제외한다.

## 2. 범위 고정

### 2.1 포함 범위

| 영역 | 이번 단계의 범위 |
| --- | --- |
| 실행 기반 | Boot, 상태 관리, 씬 전환, 오류 표시 |
| 입력 | Gameplay/UI 입력 분리, 키 설정의 최소 구조 |
| 플레이어 | 좌우 이동, 점프, 더블점프, 활공, 대시 |
| 전투 | 기본 공격, Hitbox, Damage, 피격, 사망 |
| 모듈 | 정의, 장착, 사용, 쿨타임, 최소 2~4개 모듈 |
| 퀘스트 | 순차 튜토리얼 퀘스트 8개 |
| 진행 | 경험치/레벨의 최소 동작, 모듈 포인트의 최소 지급 |
| 송신탑 | 활성화, 진행 이벤트, 저장 보조 |
| 보스 | 헬테의 최소 3패턴과 처치 보상 |
| 저장 | 영구 데이터와 회차 데이터 분리, 튜토리얼 완료 저장 |
| UI | HUD, 현재 퀘스트, 모듈 슬롯, 보스 HP, 결과/완료 화면 |
| 검증 | 핵심 시스템 EditMode/PlayMode 테스트와 수동 플레이 체크 |

### 2.2 이번 단계에서 제외

- CHAPTER_01 STG-001~006의 전체 콘텐츠
- 미노타우르스와 최종 보스의 완성 구현
- 최종 아트, 컷신, 고급 카메라 연출
- 복잡한 모듈 트리 전체와 다수의 효과 조합
- 완성형 설정 메뉴와 다국어 시스템
- 모바일/콘솔 입력 대응
- 최적화 수치 확정과 최종 배포 빌드

제외 항목은 구조를 막지 않는 placeholder만 허용한다.

## 3. 성공 기준

다음 기준을 모두 만족하면 “기능부 및 튜토리얼 1차 완료”로 판단한다.

1. 새 실행에서 Boot를 거쳐 튜토리얼에 진입할 수 있다.
2. 플레이어가 이동, 점프, 대시, 공격을 수행할 수 있다.
3. 공격이 적에게 실제 Damage를 전달하고 적이 사망한다.
4. 플레이어 사망 시 튜토리얼 정책에 따라 재시작된다.
5. 모듈을 장착하고 사용하며 쿨타임을 확인할 수 있다.
6. 퀘스트 8개가 정해진 순서로 진행되고 완료된다.
7. 송신탑 활성화가 퀘스트와 저장 이벤트를 발생시킨다.
8. 헬테의 기본 패턴을 보고 회피하거나 공격할 수 있다.
9. 헬테 처치 후 `TREE-BOSS-HELTE`가 영구 데이터에 기록된다.
10. 튜토리얼 완료 후 CHAPTER_01 진입 지점으로 이동한다.
11. 종료 후 재실행해도 튜토리얼 완료와 보스 트리 해금이 유지된다.
12. 핵심 테스트와 수동 검증에서 치명적 오류가 없다.

## 4. 전체 진행 구조

```txt
Phase A. 프로젝트 기반 고정
    ↓
Phase B. 기능부 수직 슬라이스
    ↓
Phase C. 튜토리얼 콘텐츠 연결
    ↓
Phase D. 저장/재시작/완료 검증
```

기존 로드맵의 Save, Addressables, Validator는 삭제하지 않는다. 다만 모든 기능을 만든 뒤 적용하지 않고, 이번 범위에서 필요한 최소 형태를 먼저 구현한다.

## 5. Phase A - 프로젝트 기반 고정

예상 기간: 3~5일

### 작업

| 순서 | 작업 | 완료 산출물 |
| --- | --- | --- |
| A-01 | Unity 버전, URP 2D Renderer, 입력 패키지 확인 | 프로젝트 기준 기록 |
| A-02 | `Assets/_Project` 하위 폴더 생성 | Core, Gameplay, Content, UI, Scenes, Tests |
| A-03 | asmdef 최소 구성 | Core, Gameplay, Content, UI, Tests |
| A-04 | Boot, Loading, Tutorial 씬 생성 | Build Settings 등록 |
| A-05 | ServiceRoot 생성 | 중복 생성 방지 |
| A-06 | GameState 최소 구현 | Booting, Loading, Tutorial, Paused, Result, Error |
| A-07 | GameEventBus와 GameClock 구현 | 타입 기반 이벤트, 일시정지 시간 분리 |
| A-08 | 개발용 로그/오류 화면 추가 | 오류 발생 위치 확인 가능 |

### 완료 기준

- Boot 씬에서 ServiceRoot가 한 번만 생성된다.
- Loading 씬을 거쳐 Tutorial 씬이 열린다.
- 컴파일 오류와 asmdef 순환 참조가 없다.
- 치명적 오류가 발생하면 Error 상태로 이동한다.

### 이 단계에서 만들지 않을 것

- Addressables 전체 그룹 설계
- 완성형 콘텐츠 레지스트리
- 최종 Validator 전부

초기에는 씬과 데이터의 연결을 확인할 정도의 최소 로더만 둔다. 기반이 확인된 뒤 Addressables와 Validator를 확장한다.

## 6. Phase B - 기능부 수직 슬라이스

예상 기간: 2~3주

이 단계의 목표는 아트가 없어도 하나의 테스트 방에서 플레이어가 공격하고, 적을 처치하고, 모듈을 쓰고, 죽었다가 다시 시작하는 것이다.

### B-1. 입력과 플레이어 이동

| 작업 | 기준 |
| --- | --- |
| Input Action | Move, Jump, Dash, Attack, Module, Pause |
| 입력 모드 | Gameplay와 UI 입력 분리 |
| 이동 | Rigidbody2D 기반 |
| 점프 | 지면 점프, 더블점프, 가변 점프 |
| 감각 | coyote time, jump buffer |
| 대시 | 거리, 지속 시간, 쿨타임, 짧은 무적 |
| 활공 | 공중 낙하 속도 제한 |
| 테스트 | 벽 끼임, 낙하, 연속 점프, 대시 중 입력 확인 |

### B-2. 전투 기본부

| 작업 | 기준 |
| --- | --- |
| Actor 구조 | Player, Enemy, Boss 공통 생명주기 |
| 판정 | Hurtbox, Hitbox, Projectile 최소 지원 |
| 피해 | DamagePayload로 전달 |
| 효과 | Knockback, Stun의 최소 구현 |
| 공격 | 기본 근접 공격 1종 |
| 피격 | 피격 경직, 무적 시간 |
| 사망 | Death 이벤트 1회 발생 |
| 디버그 | Hitbox/Hurtbox Gizmo와 피해 로그 |

Z축은 렌더링 정렬에만 사용하고, 실제 전투 판정은 X/Y Physics2D로 고정한다.

### B-3. 최소 데이터 구조

```txt
Definition
→ 고정 설계 데이터

RuntimeState
→ 현재 플레이 상태

SaveData
→ 영구 데이터 / 회차 데이터 / 설정 데이터

Executor
→ Definition을 실제 게임 동작으로 변환
```

최초 구현 Definition은 다음으로 제한한다.

- PlayerMotorDefinition
- WeaponDefinition
- ModuleDefinition
- EnemyDefinition
- BossDefinition
- QuestDefinition
- RewardDefinition
- TutorialStageDefinition

### B-4. 최소 모듈 시스템

첫 구현에서는 모듈 트리 전체를 만들지 않는다. 아래 모듈로 실행 파이프라인을 먼저 검증한다.

| 모듈 | 목적 |
| --- | --- |
| 기본 근접 강화 | 공격 Damage 변경 |
| 증기탄 | Projectile 생성 |
| 추진 이동 | 짧은 이동 보정 |
| 순간 재진입 | 짧은 대시 또는 블링크 |

모듈 실행 순서는 다음과 같다.

```txt
Input
→ ModuleSlot 확인
→ 사용 가능 상태 확인
→ AbilityExecutor 실행
→ Hitbox/Projectile/VFX 생성
→ Cooldown 시작
→ ModuleUsed 이벤트
```

### B-5. 회차 최소 루프

```txt
Tutorial 시작
→ 전투
→ PlayerDead
→ Tutorial 재시작 (완료 퀘스트와 송신탑 점령은 유지, 전투 주체 HP/상태만 초기화)
```

이 단계에서는 CHAPTER_01 전체 재시작보다 튜토리얼 전용 재시작을 먼저 닫는다. 이후 공통 `RunResetService`로 확장한다.

### Phase B 완료 기준

- 테스트 방에서 10분 이상 반복 플레이 가능하다.
- 플레이어가 적을 공격하고 적이 사망한다.
- 모듈 2개 이상이 서로 다른 동작을 수행한다.
- 플레이어 사망 후 완료 퀘스트와 송신탑 점령은 유지한 채, 플레이어와 활성 전투 대상이 중복 없이 초기화된다.
- 입력, 이동, 전투, 모듈의 핵심 PlayMode 테스트가 통과한다.

## 7. Phase C - 튜토리얼 콘텐츠 연결

예상 기간: 1~2주

기능부가 닫힌 뒤 튜토리얼을 붙인다. 튜토리얼 퀘스트가 코드를 직접 제어하지 않도록 이벤트를 통해 연결한다.

### C-1. 튜토리얼 진행 순서

| 순서 | 퀘스트 | 완료 이벤트 |
| --- | --- | --- |
| 1 | 이동하기 | MovementPerformed |
| 2 | 점프하기 | JumpPerformed |
| 3 | 기본 공격하기 | AttackPerformed |
| 4 | 대시하기 | DashPerformed |
| 5 | 모듈 사용하기 | ModuleUsed |
| 6 | 모듈 트리 확인하기 | ModuleTreeOpened |
| 7 | 송신탑 활성화 | RelayActivated |
| 8 | 헬테 처치 | BossKilled |

### C-1.1. 100m 튜토리얼 공간 배치

| 구간 | X 위치 | 사전 배치 요소 |
| --- | ---: | --- |
| 시작 | -42m | PlayerRoot, TutorialSpawnPoint |
| 전투 | -25m | TutorialEnemy |
| 송신탑 | -4m | TutorialRelay |
| 송신탑 체크포인트 | 0m | TutorialRelayCheckpoint |
| 보스 | +30m | TutorialHelte, GoalMarker |
| 지형 | -50m ~ +50m | TerrainBase, 선택형 발판 4개, 양끝 2D 경계 |

카메라는 PlayerRoot를 X축으로 추적하며, 지형의 100m 폭을 넘지 않도록 제한한다. 발판은 주 경로를 차단하지 않는 선택형 이동 요소다.

송신탑을 활성화하면 `TutorialRestartHost`의 재시작 지점이 사전 배치된 `TutorialRelayCheckpoint`로 전환된다. 이후 사망은 완료 퀘스트와 송신탑 점령을 유지한 채 체크포인트에서 재개하며, 카메라도 재시작 위치로 즉시 스냅된다.

송신탑 상호작용은 `QST-TUTO-007`이 `InProgress`일 때만 허용한다. 이 게이트는 맵을 먼저 탐색했을 때 송신탑 이벤트가 조기에 소비되어 퀘스트가 막히는 것을 방지한다.

### C-2. 튜토리얼 방 구성

| 방 | 내용 | 필요한 시스템 |
| --- | --- | --- |
| TUTO-001 | 이동/점프 | Input, Movement |
| TUTO-002 | 대시/활공 | Movement, Cooldown |
| TUTO-003 | 기본 공격 | Combat, Enemy |
| TUTO-004 | 모듈 사용 | Module, HUD |
| TUTO-005 | 모듈 트리/송신탑 | UI, Save, Quest |

### C-3. 모듈 트리 적용 현황

- `ModuleTreeDefinition`, `ModuleNodeDefinition`, `ModuleTreeManager`로 트리 접근 권한, 선행 노드, 모듈 포인트, 회차 해금/장착 상태를 분리했다.
- `I` 입력으로 사전 배치된 `TutorialHUD/ModuleTreePanel`을 열고 `ModuleTreeOpened(TREE-BASIC-001)` 이벤트를 발행한다.
- `QST-TUTO-006`은 위 이벤트로 완료되며 다음 목표인 `QST-TUTO-007` 송신탑 활성화로 진행한다.
- `StageSystems`의 `ModuleTreeManagerHost`와 `TutorialHUD`의 `ModuleTreePanelPresenter`은 Inspector 참조로 연결되어 있으며 런타임에 UI/GameObject를 생성하지 않는다.
| TUTO-006 | 헬테 아레나 | Boss, Camera, Sound |
| TUTO-007 | 결과/CHAPTER_01 진입 | Result, SceneFlow |

처음에는 여러 씬으로 분리하지 않고 하나의 Tutorial 씬 안에서 RoomRoot를 전환하는 방식으로 빠르게 검증할 수 있다. 씬 전환이 안정화된 뒤 필요하면 Addressables 씬으로 분리한다.

### C-4. 헬테 최소 범위

| 패턴 | 동작 |
| --- | --- |
| `PAT-HELTE-BASIC-01` | 근접 Hitbox를 0.18초 간격으로 2회 활성화, 각 8 피해 |
| `PAT-HELTE-BLINK-DASH-01` | 사전 배치된 좌/우 안전 Anchor로 이동한 뒤 0.35초 예고 후 X자 Hitbox를 1회 활성화, 15 피해. 이동 자체에는 피해가 없다. |
| `PAT-HELTE-SUMMON-SWORD-01` | 사전 배치된 칼 Hitbox 3개를 발사 시점의 플레이어 위치에 순차 배치해 각 10 피해를 준다. |

필수 조건은 패턴 수가 아니라 다음 네 가지다.

- 패턴 예고가 보인다.
- Hitbox 위치와 시각 효과가 일치한다.
- 플레이어가 회피할 수 있다.
- 보스 처치 이벤트가 한 번만 발생한다.

### C-4 적용 현황

- `HeltePatternPlanner`는 일반 공격 2회 뒤 1페이즈에서 블링크, 2페이즈에서 칼 소환 후 블링크를 순서대로 선택한다.
- `TutorialHelte`에는 `HelteBossPatternHost`와 `BlinkLeftAnchor`, `BlinkRightAnchor`, 기본/X자/칼 3개 Hitbox가 모두 계층에 사전 배치되어 있다.
- 기존 `EnemyAttackHost`는 헬테에서 비활성화했다. 전용 패턴 호스트만 공격 판정을 제어한다.
- 런타임에는 GameObject/Component를 생성하지 않으며, 칼 판정은 이미 배치된 3개 Collider의 위치만 갱신한다.

### Phase C 완료 기준

- 8개 퀘스트가 순서대로 진행된다.
- 현재 목표가 HUD에 표시된다.
- 퀘스트 완료 후 다음 퀘스트가 자동 시작된다.
- 헬테 처치 후 보스 모듈 트리가 해금된다.
- 튜토리얼 완료 후 CHAPTER_01 진입 placeholder가 열린다.

### C-5. 종료 루프 적용 현황

- `QST-TUTO-007` 완료 시 `TutorialBossEncounterHost`가 사전 배치된 `TutorialHelte`를 활성화하고 `QST-TUTO-008`을 시작한다.
- 보스 전투 시스템은 `BossKilled`와 `GameplaySignal(BossKilled)`를 함께 발행하여 보상과 퀘스트 완료 경로를 분리한다.
- `TutorialBossCompletionHost`는 헬테 처치 한 번만 처리하며 `TutorialCompleted`, `TREE-BOSS-HELTE` 영구 해금, 보스 처치 기록, `RunData.CurrentStageId = CHAPTER_01`을 저장한다.
- `CHAPTER_01`은 완료 결과 오버레이의 사전 배치된 진입 버튼으로 최소 진입 씬에만 연결한다. 버튼은 튜토리얼 완료 저장과 `RunData.CurrentStageId = CHAPTER_01`이 모두 확인될 때만 전환한다. CHAPTER_01의 콘텐츠 제작은 튜토리얼 완료 검증 이후 단계로 보류한다.

## 8. Phase D - 저장, 재시작, 검증

예상 기간: 3~5일

### 저장 구분

| 구분 | 유지 내용 |
| --- | --- |
| PermanentData | 튜토리얼 완료, 보스 모듈 트리 해금, 누적 사망 횟수 |
| RunData | 현재 퀘스트, 현재 HP, 임시 모듈, 경험치, 모듈 포인트 |
| SettingsData | 음량, 화면, 키 설정 |

현재 튜토리얼은 `RunData.QuestIds`에 완료 퀘스트를 저장한다. 재실행 시 첫 미완료 퀘스트부터 재개한다. 송신탑 점령 상태, 보스 활성 상태, 송신탑 체크포인트는 `ActivatedTowerIds`의 송신탑 ID와 `QST-TUTO-007` 완료 기록이 모두 있을 때만 함께 복원한다. 저장 도중 종료되어 송신탑 ID만 남은 불완전 데이터는 점령 전 상태로 재개해 퀘스트 진행을 막지 않는다.

### 검증 시나리오

1. 새 게임으로 튜토리얼에 진입한다.
2. 튜토리얼 중 강제 종료 후 다시 실행한다.
3. 튜토리얼 완료 여부와 회차 데이터의 차이를 확인한다.
4. 헬테 처치 후 저장한다.
5. 재실행해 `TREE-BOSS-HELTE`가 유지되는지 확인한다.
6. 새 회차를 시작해 임시 모듈과 퀘스트가 초기화되는지 확인한다.
7. 중복 이벤트, 중복 보상, 중복 보스 처치가 없는지 확인한다.

### 이 단계에서 만들 최소 검증기

- Stable ID 중복 검사
- 필수 참조 누락 검사
- Save DTO 직렬화 검사
- Tutorial Quest 연결 검사
- Boss Reward 연결 검사
- Build Settings 씬 검사

전체 콘텐츠 Validator는 CHAPTER_01 제작 시작 전에 확장한다.

## 13. 최종 판단

이 로드맵의 난이도는 중상급이지만, 현재 프로젝트의 목표와 기간을 통제할 수 있다면 충분히 실행 가능하다.

가장 중요한 기준은 “시스템을 전부 만들고 나중에 튜토리얼을 붙이는 것”이 아니다. 기능부를 작은 수직 슬라이스로 닫은 직후 튜토리얼에 연결하여, 데이터 구조와 플레이 감각이 실제 흐름에서 맞는지 바로 확인하는 것이다.

이번 목표가 완료되면 다음 단계는 CHAPTER_01 전체가 아니라, 먼저 `STG-001 → 일반 적 → 포탈 → 미노타우르스`를 확장하는 별도 로드맵으로 분리한다.
