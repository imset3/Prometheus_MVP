# 32. 시스템 아키텍처 설계 초안

## 1. 설계 목적

이번 단계에서는 화면이나 콘텐츠를 먼저 만들지 않는다. 게임이 실행되고, 데이터를 읽고, 상태를 변경하고, 저장하고, 씬을 이동하는 내부 구조를 먼저 확정한다.

아키텍처의 최우선 목표는 다음 네 가지다.

1. 각 시스템의 책임이 겹치지 않아야 한다.
2. 콘텐츠 ID를 코드에 직접 하드 코딩하지 않아야 한다.
3. 영구 데이터와 회차 데이터를 혼동하지 않아야 한다.
4. 나중에 튜토리얼과 CHAPTER_01을 같은 구조로 확장할 수 있어야 한다.

## 2. 아키텍처의 최상위 구조

```txt
┌──────────────────────────────────────────────┐
│ Presentation                                  │
│ UI / Camera / Audio / VFX / Animation Bridge  │
└───────────────────────┬──────────────────────┘
                        │ 이벤트 구독 / ViewModel
┌───────────────────────▼──────────────────────┐
│ Gameplay                                     │
│ Player / Combat / Module / Quest / Boss       │
└───────────────┬───────────────────┬──────────┘
                │ Definition 읽기    │ RuntimeState 변경
┌───────────────▼──────────────┐    │
│ Content                      │    │
│ Registry / Definition /      │    │
│ Validation                   │    │
└───────────────┬──────────────┘    │
                │                    │
┌───────────────▼────────────────────▼─────────┐
│ Core                                         │
│ Bootstrap / State / EventBus / Clock / Scope  │
└───────────────┬──────────────────────────────┘
                │
┌───────────────▼──────────────────────────────┐
│ Save + SceneFlow                              │
│ DTO / Migration / Load / Transition / Restart │
└───────────────────────────────────────────────┘
```

핵심 의존 방향은 아래와 같다.

```txt
Core ← 모든 런타임 계층이 사용 가능
Content ← Gameplay가 읽기 전용으로 사용
Save ← RuntimeState를 DTO로 변환
SceneFlow ← Core, Content, Save를 조합
Presentation ← Gameplay 이벤트를 구독
Gameplay → Presentation 직접 참조 금지
```

## 3. 레이어별 책임

| 레이어 | 책임 | 소유하면 안 되는 것 |
| --- | --- | --- |
| Core | 앱 수명, 상태, 시간, 이벤트, 공통 오류 | 콘텐츠 규칙, UI 표시, 저장 파일 형식 |
| Content | Definition 로드, ID 검색, 에셋 검증 | 런타임 진행도, 플레이어 HP, 씬 이동 |
| Save | DTO 저장/로드, 버전 변환, 원자적 쓰기 | 전투 판정, UI, 씬 오브젝트 |
| SceneFlow | Boot/Loading/Stage/Restart/Result 흐름 | 퀘스트 조건, 보스 패턴, UI 세부 구현 |
| Gameplay | 실제 게임 규칙과 RuntimeState 변경 | ScriptableObject 수정, 파일 직접 저장, UI 직접 조작 |
| Presentation | 이벤트를 화면과 연출로 변환 | 게임 규칙 판정, 영구 데이터 수정 |
| Tools | 에디터 검증, 디버그 명령, 테스트 지원 | 런타임에서 호출되는 Editor API |

### 3.1 Hierarchy 사전 배치 원칙

Unity `GameObject`, `Component`, 시스템 Host, 스테이지 기물은 코드에서 런타임 생성하거나 부착하지 않는다.

```txt
씬/Prefab에 사전 배치
→ Inspector 참조 연결
→ Host Component가 초기화
→ 순수 RuntimeState만 변경
```

`new GameObject`, `Instantiate`, `AddComponent`는 Runtime asmdef에서 금지한다. 순수 C# RuntimeState, DTO, 이벤트 payload는 Unity 오브젝트가 아니므로 생성할 수 있다.

## 4. 프로젝트 어셈블리 구조

초기에는 아래 8개 asmdef를 만든다. 시스템이 안정된 후 세분화한다.

```txt
Narthex.Core
    └─ UnityEngine 최소 의존

Narthex.Content
    └─ Narthex.Core

Narthex.Save
    └─ Narthex.Core, Narthex.Content

Narthex.SceneFlow
    └─ Narthex.Core, Narthex.Content, Narthex.Save

Narthex.Gameplay
    └─ Narthex.Core, Narthex.Content, Narthex.Save

Narthex.Presentation
    └─ Narthex.Core, Narthex.Content, Narthex.Gameplay

Narthex.Tests
    └─ 모든 런타임 asmdef, Test Framework

Narthex.Tools (Editor only)
    └─ Narthex.Core, Narthex.Content, Narthex.Save, Narthex.Gameplay
```

Editor 전용 검증은 별도 asmdef로 분리한다.

```txt
Narthex.Tools
    └─ Narthex.Core, Narthex.Content, Narthex.Save, Narthex.Gameplay
    └─ UnityEditor API 허용
```

Runtime asmdef가 `Narthex.Editor` 또는 `UnityEditor`를 참조하면 구조 검증 실패로 처리한다.

## 5. Core 설계

### 5.1 Bootstrap

`GameBootstrap`은 앱이 시작될 때 다음 초기화 순서를 보장한다.

```txt
Bootstrap 시작
→ ServiceRoot 생성/중복 제거
→ GameClock 생성
→ GameEventBus 생성
→ ContentRegistry 로드
→ Content 검증
→ SaveSystem 초기화
→ GameState를 Title 또는 Error로 전환
```

Bootstrap은 플레이어, 적, 보스, 퀘스트를 직접 생성하지 않는다.

### 5.2 GameState

```txt
Booting
→ Loading
→ Title
→ Tutorial
→ InRun
→ Paused
→ Result
→ Error
```

상태 전환은 `GameStateMachine` 하나만 수행한다. 개별 시스템이 `SceneManager.LoadScene` 또는 `Time.timeScale`을 직접 호출하지 않는다.

### 5.3 EventBus

이벤트 버스는 `ServiceRoot`가 소유하는 인스턴스 방식으로 구현한다.

금지 사항:

- static event
- 문자열 이벤트명
- UI 오브젝트 직접 호출
- 이벤트 핸들러 안에서 예외를 숨기는 방식

이벤트는 사실을 알리고, 수신 시스템이 자신의 책임에 맞게 반응한다.

### 5.4 GameClock

시간은 아래 세 종류로 구분한다.

| 시간 | 사용처 |
| --- | --- |
| ScaledTime | 일반 게임플레이, 쿨타임 |
| UnscaledTime | Pause 중에도 작동하는 UI와 입력 |
| FixedTime | Rigidbody2D 이동과 물리 판정 |

Gameplay는 `Time.time`을 직접 사용하지 않고 `GameClock`을 사용한다.

## 6. Content 설계

### 6.1 Definition과 RuntimeState

```txt
Definition Asset
    고정 데이터
    ScriptableObject
    런타임에서 수정 금지

RuntimeState
    현재 회차의 변경 데이터
    순수 C# 클래스 또는 struct
    시스템이 변경

Save DTO
    RuntimeState의 저장용 표현
    단순 타입과 ID만 보유
```

### 6.2 ContentRegistry

ContentRegistry는 Definition을 찾는 검색 원장이다. Gameplay 시스템은 씬에서 `FindObjectOfType` 또는 직접적인 에셋 경로 검색을 사용하지 않는다.

```txt
ContentRegistry
→ stableId 검증
→ Definition 검색
→ 필수 참조 검증
→ Addressable 주소 검증
```

### 6.3 초기 Definition 목록

첫 구조에서는 다음 Definition만 만든다.

- `PlayerMotorDefinition`
- `WeaponDefinition`
- `ModuleDefinition`
- `AbilityDefinition`
- `ActionDefinition`
- `EffectDefinition`
- `EnemyDefinition`
- `BossDefinition`
- `BossPatternDefinition`
- `QuestDefinition`
- `QuestConditionDefinition`
- `RewardDefinition`
- `StageDefinition`
- `RoomDefinition`
- `PortalDefinition`
- `TowerDefinition`
- `CameraBoundsDefinition`
- `BalanceProfileDefinition`
- `HitboxDefinition`

## 7. Save 설계

### 7.1 데이터 경계

| 데이터 | 저장 영역 | 회차 재시작 시 |
| --- | --- | --- |
| 튜토리얼 완료 | PermanentData | 유지 |
| 보스 모듈 트리 해금 | PermanentData | 유지 |
| 누적 사망 횟수 | PermanentData | 유지 |
| 현재 스테이지 | RunData | 초기화 |
| 레벨/경험치 | RunData | 초기화 |
| 모듈 장착/강화 | RunData | 초기화 |
| 퀘스트 진행도 | RunData | 초기화 |
| 송신탑 상태 | RunData | 초기화 |
| 음량/화면/키 설정 | SettingsData | 유지 |

### 7.2 저장 흐름

```txt
RuntimeState 변경
→ SaveRequested 이벤트 또는 명시적 저장 요청
→ SaveMapper가 DTO 생성
→ SaveSerializer가 JSON 생성
→ 임시 파일 기록
→ 파일 검증
→ 기존 파일 교체
→ SaveCompleted 또는 SaveFailed 이벤트
```

저장 시스템은 Gameplay를 직접 호출하지 않는다. Gameplay가 저장 요청을 발생시키고 Save가 현재 상태를 읽어 DTO로 변환한다.

금지 타입:

- `UnityEngine.Object`
- `ScriptableObject`
- `Dictionary`
- `SceneAsset`
- 인터페이스 또는 추상 클래스 직접 저장

## 8. SceneFlow 설계

SceneFlow는 모든 씬 이동의 단일 진입점이다.

```txt
NewGame
Continue
EnterTutorial
EnterStage
UsePortal
RestartTutorial
RestartRun
ShowResult
```

흐름은 다음과 같다.

```txt
SceneFlow 요청
→ 현재 상태 전환
→ 필요한 Save 처리
→ Loading 화면 표시
→ Addressable/씬 로드
→ RuntimeScope 생성
→ StageRuntime 구성
→ Gameplay 활성화
→ 완료 이벤트 발행
```

씬 오브젝트가 다른 씬을 직접 로드하지 않는다.

### 8.1 회차 재시작 책임

`RunRestartCoordinator`가 `PlayerDead`를 단일 수신하고 다음 순서를 조정한다.

```txt
PlayerDead
→ PermanentSaveData.TotalDeaths 증가
→ RunContext.Reset(startStageId)
→ RunRestarted 발행
→ SceneFlow가 지정된 시작 스테이지 로드
```

튜토리얼 중 사망은 튜토리얼 시작 지점을 사용하고, 튜토리얼 완료 후의 회차 사망은 `STG-001`을 사용한다. 송신탑은 재시작 지점을 소유하지 않는다.

## 9. Gameplay 실행 구조

### 9.1 공통 실행 파이프라인

모듈, 적 공격, 보스 패턴은 같은 실행 파이프라인을 사용한다.

```txt
입력 또는 AI 결정
→ AbilityRequested
→ AbilityDefinition 조회
→ AbilityExecutor 실행
→ ActionDefinition 순차 처리
→ Hitbox/Projectile 생성
→ CombatSystem 명중 확인
→ EffectSystem 적용
→ 결과 이벤트 발행
```

### 9.2 Combat 책임

CombatSystem은 아래만 담당한다.

- Hitbox/Hurtbox 탐지
- 중복 명중 방지
- DamagePayload 계산 요청
- 피격/사망 이벤트 발행

CombatSystem은 퀘스트 보상이나 UI를 직접 처리하지 않는다.

### 9.3 Module 책임

ModuleSystem은 아래만 담당한다.

- 모듈 해금 상태 확인
- 장착 슬롯 관리
- 사용 가능 여부 확인
- Ability 실행 요청
- 쿨타임과 회차 상태 관리

ModuleSystem은 Ability 내부의 Hitbox 세부 동작을 알지 못한다.

모듈 발동 위치와 연출은 `DeusCastAnchor`와 `ModuleCastPolicy`가 담당한다.

```txt
ModuleCastPolicy
→ DeusFront
→ PlayerCenter
→ TargetPosition
→ CustomAnchor
```

일반 공격은 모듈이 아니며, 별도의 `BasicAttackRequest`와 무기 실행 계약으로 처리한다.

### 9.4 Quest 책임

QuestManager는 이벤트를 받아 진행도와 상태만 변경한다.

```txt
GameEventBus 이벤트
→ QuestConditionEvaluator
→ QuestRuntimeState 갱신
→ QuestCompleted
→ RewardExecutor 요청
```

QuestManager는 UI, SaveSystem, ModuleSystem을 직접 호출하지 않는다.

## 10. 이벤트 계약

| 이벤트 | 발생 시스템 | 주요 수신 시스템 |
| --- | --- | --- |
| `PlayerDead` | Combat | SceneFlow, Save, UI, Audio |
| `EnemyKilled` | Combat | Level, Quest, UI, Audio |
| `BossKilled` | Combat | Quest, Reward, Save, UI |
| `ModuleUsed` | Module | Quest, UI, Audio |
| `TowerActivated` | Stage | Quest, Save, UI |
| `QuestCompleted` | Quest | Reward, Save, UI |
| `RewardGranted` | Reward | Save, ModuleTree, UI |
| `PortalUsed` | Portal | SceneFlow, Quest, Save |
| `RunRestarted` | SceneFlow | Save, Gameplay, UI |

이벤트 payload에는 수신자가 판단에 필요한 ID와 최소 상태만 포함한다. Unity 오브젝트 참조를 이벤트 payload에 넣지 않는다.

하위 실행 이벤트는 다음과 같이 분리한다.

```txt
AbilityRequested
AbilityExecuted
HitConfirmed
EffectApplied
PlayerHit
StatusEffectApplied
SaveCompleted
SaveFailed
```

## 11. Runtime 수명과 소유권

| 객체 | 소유자 | 수명 |
| --- | --- | --- |
| GameEventBus | ServiceRoot | 앱 전체 |
| ContentRegistry | ServiceRoot | 앱 전체 |
| SaveSystem | ServiceRoot | 앱 전체 |
| RunState | RunContext | 회차 전체 |
| StageRuntime | SceneFlow | 현재 스테이지 |
| BossRuntime | StageRuntime | 보스방 |
| QuestRuntime | RunContext | 회차 전체 |
| AddressableLoadScope | SceneFlow/StageRuntime | 소유 씬 또는 기능 |
| Hitbox/Projectile | PoolService | 생성부터 반환까지 |
| RunRestartCoordinator | SceneFlow | PlayerDead부터 새 회차 시작까지 |
| DeusCastAnchor | DeusRuntime | 데우스 발동 위치와 연출 기준 |

소유자가 명확하지 않은 객체는 생성하지 않는다. 수명 종료 시 구독 해제, 핸들 해제, 풀 반환이 함께 일어나야 한다.

## 12. 구현 순서

아키텍처 구현은 다음 순서로 진행한다.

### A. 계약만 먼저 작성

- 폴더와 asmdef
- 공통 ID 타입
- GameState enum
- 이벤트 payload 타입
- Save DTO 타입
- DefinitionBase

### B. Core 런타임

- ServiceRoot
- GameStateMachine
- GameEventBus
- GameClock
- ErrorState

### C. Content/Save

- ContentRegistry
- SaveMapper
- SaveSerializer
- MigrationRunner
- 기본 검증기

### D. SceneFlow

- Boot 흐름
- Loading 흐름
- RuntimeScope
- Tutorial 진입과 재시작 요청

### E. Gameplay 계약 구현

- AbilityExecutor
- CombatSystem 인터페이스와 최소 구현
- ModuleSystem 상태 관리
- QuestManager 상태 관리
- RewardExecutor
- RunRestartCoordinator
- DeusCastAnchor / ModuleCastPolicy
- BasicAttackRequest
- RoomDefinition / PortalDefinition / TowerDefinition

이 순서가 끝난 뒤에야 플레이어 오브젝트, 적, 보스, UI를 연결한다.

## 13. 아키텍처 완료 기준

다음 조건을 만족하면 시스템 아키텍처 1차 완료로 판단한다.

1. Runtime asmdef가 Editor API 없이 컴파일된다.
2. GameStateMachine 외부에서 게임 상태를 변경하지 않는다.
3. SceneFlow 외부에서 씬을 직접 로드하지 않는다.
4. Gameplay가 ScriptableObject Definition을 수정하지 않는다.
5. Save DTO에 Unity 객체와 금지 타입이 없다.
6. PermanentData와 RunData의 초기화 규칙이 테스트로 확인된다.
7. 이벤트 payload가 ID 중심이며 순환 참조가 없다.
8. ContentRegistry에서 Definition을 stableId로 검색할 수 있다.
9. Ability 실행을 모듈과 보스가 공유할 수 있다.
10. 퀘스트가 UI를 직접 호출하지 않고 이벤트로 진행된다.
11. 씬 전환과 회차 재시작 요청을 코드 한 경로로 처리할 수 있다.
12. 핵심 시스템의 단위 테스트가 실행된다.
13. 회차 재시작이 영구 데이터 유지와 회차 데이터 초기화를 함께 검증한다.
14. 데우스 발동 위치와 기본 공격 계약이 모듈 계약과 분리되어 있다.
15. 모든 시스템 Host와 스테이지 오브젝트가 Hierarchy 또는 Prefab에 사전 배치되어 있다.
16. Runtime 코드에 `new GameObject`, `Instantiate`, `AddComponent`가 없다.

## 14. 아키텍처 단계에서 보류할 것

- 화면 레이아웃
- 최종 플레이어 스프라이트
- 보스 연출 카메라
- 다수의 모듈 콘텐츠
- 최종 사운드와 VFX
- CHAPTER_01 전체 씬 제작

이 항목들은 아키텍처 완료 기준을 통과한 뒤 실제 이벤트와 RuntimeState에 연결한다.
