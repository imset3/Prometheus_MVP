# 33. 계층 사전배치 및 런타임 생성 금지 정책

## 1. 목적

게임에서 동작하는 모든 Unity `GameObject`, `Component`, 시스템 Host, 스테이지 기물은 코드가 실행 중에 생성하거나 부착하지 않는다.

필요한 오브젝트와 Component는 Unity Hierarchy 또는 Prefab에 사전 배치하고, 런타임 코드는 이미 존재하는 Host의 참조를 받아 상태와 데이터만 변경한다.

## 2. 절대 규칙

| 규칙 | 기준 |
| --- | --- |
| GameObject 생성 금지 | 런타임의 `new GameObject`, `Instantiate` 사용 금지 |
| Component 부착 금지 | 런타임의 `AddComponent` 사용 금지 |
| 시스템 Host 사전 배치 | Bootstrap, Save, SceneFlow, Combat, Module, Quest, Reward Host는 씬 Hierarchy에 존재해야 함 |
| 스테이지 기물 사전 배치 | Player, Enemy, Boss, Portal, Tower, HitboxAnchor, SpawnPoint는 씬 또는 Prefab에 미리 작성 |
| Prefab은 명시적으로 배치 | 런타임 소환이 필요한 콘텐츠도 이번 MVP에서는 Stage Scene의 비활성 오브젝트 또는 사전 배치 Pool Root를 사용 |
| RuntimeState 예외 | 순수 C# 상태값, DTO, 이벤트 payload는 GameObject가 아니므로 생성 가능. Unity 오브젝트를 만들거나 부착해서는 안 됨 |

## 3. Hierarchy 기준

```txt
Boot
└─ Bootstrap
   ├─ ServiceRoot
   ├─ GameBootstrap
   ├─ SaveSystemHost
   └─ SceneFlowHost

Stage
└─ StageRoot
   ├─ GameplaySystems
   │  ├─ CombatSystemHost
   │  ├─ ModuleSystemHost
   │  ├─ ModuleTreeManagerHost
   │  ├─ QuestManagerHost
   │  ├─ RewardExecutorHost
   │  ├─ TutorialBossEncounterHost
   │  └─ TutorialBossCompletionHost
   ├─ PlayerRoot
   ├─ EnemyRoot
   ├─ BossRoot
   ├─ PortalRoot
   ├─ TowerRoot
   └─ PoolRoot
```

## 4. 구현 기준

1. Host Component는 Inspector에서 필요한 Definition, 다른 Host, Anchor를 참조한다.
2. 참조가 없으면 자동 생성하지 않고 Console 오류와 Validator 실패로 처리한다.
3. `Awake`와 `OnEnable`은 초기화와 참조 검증만 수행한다.
4. 전투 판정 오브젝트는 사전 배치 Pool Root의 비활성 인스턴스를 활성화/반환하는 방식으로 처리한다.
5. Addressables 로드는 무거운 에셋 로딩 정책이며, 런타임 Component 생성 허용 근거가 아니다.

## 5. 검증 기준

- Runtime asmdef에서 `new GameObject`, `Instantiate`, `AddComponent` 사용 시 Validator 오류
- Boot Scene에 필수 Host가 없으면 Validator 오류
- StageRoot에 필수 Systems/Roots가 없으면 Validator 오류
- Component가 Inspector 필수 참조를 잃으면 PlayMode 진입 전 오류
- Unity 메뉴 `Narthex/Validation/Validate Active Tutorial Scene`은 StageSystems, PlayerRoot, TutorialEnemy, GoalMarker, TutorialHUD와 필수 Component/Child 구조를 검증한다.

## 6. 현재 적용 상태 (2026-07-10)

`Boot.unity`에는 `Bootstrap` 아래의 `ServiceRoot`, `GameBootstrap`, `SaveSystemHost`, `SceneFlowHost`, `GameplaySystems` 및 Combat/Module/Quest/Reward Host를 사전 배치했다.

`TutorialScene.unity`에는 다음의 첫 스테이지 기반 계층을 사전 배치했다.

```txt
StageRoot
├─ StageGeometryRoot
├─ Ground (BoxCollider2D)
├─ StageSystems (ServiceRoot, CombatSystemHost)
├─ PlayerRoot
   ├─ GroundProbe
   └─ AttackAnchor
      └─ BayonetHitbox (BoxCollider2D)
└─ EnemyRoot
   └─ TutorialEnemy (BoxCollider2D, CombatActorHost, EnemyAttackHost)
      └─ EnemyAttackAnchor
         └─ EnemyMeleeHitbox (BoxCollider2D)
   └─ TutorialHelte (비활성, BoxCollider2D, CombatActorHost, EnemyAttackHost)
      ├─ BossAttackAnchor (BoxCollider2D)
      └─ BossVisual

GoalMarker
└─ GoalTrigger (BoxCollider2D)

TutorialHUD
├─ TutorialStatusText (UI Text, TutorialStatusPresenter)
├─ PlayerHealthText (UI Text, CombatHealthTextPresenter)
├─ EnemyHealthText (UI Text, CombatHealthTextPresenter)
└─ TutorialResultOverlay (Image)
   └─ TutorialResultText (UI Text)
```

`PlayerRoot`에는 `Rigidbody2D`, `CapsuleCollider2D`, `PlayerInput`, `PlayerMotorHost`, `PlayerInputHost`, `CombatActorHost`, `MeleeAttackHost`를 부착하고, Input Actions, 이동 Definition, GroundProbe, CombatSystemHost, BayonetHitbox를 Inspector 참조로 연결했다. 일반 공격은 사전 배치된 `BayonetHitbox` Collider를 활성화해 명중을 확인하며, 튜토리얼 적도 사전 배치된 `CombatActorHost`로 전투 시스템에 등록한다.

`StageSystems`의 `TutorialRestartHost`는 사전 배치된 `TutorialSpawnPoint`, `TutorialRelayCheckpoint`, Player Rigidbody2D, Player/튜토리얼 적/헬테 전투 주체를 참조해 시작과 `PlayerDead` 이벤트에서 위치와 순수 RuntimeState만 초기화한다. 재실행 시에는 `ActivatedTowerIds`와 `QST-TUTO-007` 완료 기록이 모두 있을 때만 체크포인트를 사용한다. 튜토리얼 재도전은 체크포인트 방식이므로 완료 퀘스트, 송신탑 점령, 보스 활성 상태는 되돌리지 않는다. `TutorialEncounterHost`는 첫 적 처치 뒤 튜토리얼 적만 비활성화하며, 조기 완료를 막기 위해 `GoalMarker`를 활성화하지 않는다. `TutorialBossEncounterHost`는 `QST-TUTO-007` 완료 이벤트에서 비활성 상태의 `TutorialHelte`를 활성화하고, 같은 완료 조건으로 재실행 뒤 보스 상태를 복원한다. `TutorialBossCompletionHost`는 `BossKilled(BOSS-TUTO-HELTE)`를 한 번만 처리해 영구 보상과 완료 저장을 갱신한다. `TutorialVisualRoot`는 Ground, 경계, 플레이어, 적, 목표를 기본 도형으로 구성한 현재 블록아웃이다.

지형은 `StageRoot/TerrainLayoutRoot` 아래의 사전 배치된 `TerrainBlock`으로 구성한다. 각 블록은 `Collision`의 `BoxCollider2D`와 `Visual` 큐브를 함께 참조하므로, 이동·확대 시 충돌과 블록아웃 외형이 일치한다. 새 지형은 에디터 메뉴 `Prometheus/Level Design/Terrain Layout`에서만 생성하고 배치한다. 게임 실행 중에는 지형 오브젝트나 컴포넌트를 생성하지 않는다.

튜토리얼 기본 지형은 X축 `-50m ~ +50m`의 100m 폭이다. `Main Camera`의 사전 배치된 `CameraFollowHost`는 PlayerRoot를 따라가며, 현재 시야 폭을 제외한 지형 경계 안에서만 X축 이동을 허용한다. 카메라 역시 런타임 생성하지 않는다.

`GoalMarker`의 `TutorialGoalHost`는 사전 배치된 `GoalTrigger`, Player Collider, PlayerInputHost, StageSystems의 SaveSystemHost를 참조한다. 적 처치 후 Goal 영역에서 `Interact` 입력이 완료되면 `PermanentSaveData.TutorialCompleted`를 갱신하고 저장한 뒤 `TutorialCompleted` 이벤트를 발행한다.

`TutorialHUD`의 `TutorialStatusPresenter`는 ServiceRoot 이벤트 버스를 구독해 적 처치 전, GoalMarker 이동, 튜토리얼 완료 상태를 사전 배치된 UI Text에 표시한다. 프레젠테이션은 전투와 저장 데이터를 직접 수정하지 않는다.

튜토리얼의 HUD, 모듈 트리, 결과 오버레이, CHAPTER_01 진입 버튼과 모든 Text/Button은 씬 계층에 사전 배치한다. 런타임 스크립트는 기존 참조의 문구, 활성 상태, 상호작용만 갱신하며 UI GameObject, Component, Text, Button을 생성하지 않는다. 플레이어 노출 기본 문구는 한국어를 사용한다.

`PlayerHealthText`와 `EnemyHealthText`의 `CombatHealthTextPresenter`는 각 CombatActorHost의 RuntimeState를 읽어 현재 HP와 최대 HP만 표시한다. Presenter는 피해 계산, 사망 처리, 저장을 수행하지 않는다.

`StageSystems`의 `TutorialCompletionFlowHost`는 사전 배치된 SaveSystemHost를 통해 `PermanentSaveData.TutorialCompleted`를 확인한다. 미완료면 TutorialScene 시작 시 GameState를 Tutorial로 전환하고, 완료 저장이 있으면 Result 상태와 사전 배치된 `TutorialResultOverlay`를 활성화한다. `TutorialCompleted` 이벤트도 같은 Result 경로를 사용한다. 결과 오버레이와 텍스트는 런타임에 생성하지 않는다.

## 7. 적용 범위

이 정책은 튜토리얼, CHAPTER_01, 보스전, UI, VFX, Audio를 포함한 이후 모든 구현에 적용한다.
