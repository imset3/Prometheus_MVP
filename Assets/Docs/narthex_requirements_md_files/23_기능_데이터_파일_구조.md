# 23. 기능 / 데이터 / 파일 구조

## 1. 목적

이 문서는 Unity 6.3 URP 프로젝트에서 실제 구현자가 따라야 할 기능 계층, 데이터 계층, 파일 구조를 정립한다.

기준은 `콘텐츠는 Definition Asset`, `실행은 Runtime Executor`, `저장은 stableId와 RuntimeState`, `무거운 리소스는 Addressables`이다.

## 2. 기능 계층

| 계층 | 역할 | 포함 기능 | 직접 참조 가능 대상 |
| --- | --- | --- | --- |
| Core | 게임 실행 기반 | Bootstrap, GameStateMachine, EventBus, ServiceLocator, Clock | 다른 Core 서비스 |
| Content | 고정 데이터 로딩과 검증 | ContentRegistry, DefinitionDatabase, AssetValidator | Definition Asset |
| Save | 저장/로드/마이그레이션 | PermanentData, RunData, SettingsData, MigrationTable | ID와 RuntimeState |
| SceneFlow | 씬과 스테이지 전환 | Loading, StageLoad, PortalTransition, RunRestartLoad | Addressables SceneReference |
| Gameplay | 실제 플레이 로직 | Player, Movement, Combat, Module, Effect, Enemy, Boss, Quest | Definition과 RuntimeState |
| Presentation | 표시와 연출 | UI, Camera, Audio, VFX, Animation Bridge | 이벤트와 ViewModel |
| Tools | 에디터 도구 | ID Generator, Validator, Content Browser, Build Check | 에디터 전용 API |
| Tests | 검증 | EditMode, PlayMode, BuildValidation | 테스트 Fixture |

## 3. 핵심 런타임 흐름

```txt
Boot Scene
→ GameBootstrap 초기화
→ GameContentRegistry 로드
→ Definition ID / Addressables / URP 호환성 검증
→ SaveSystem 로드 또는 신규 생성
→ Title / NewGame / Continue 선택
→ Loading Scene
→ StageDefinition 기준으로 필요한 Addressables Preload
→ Stage Scene 진입
→ RunState 생성
→ Quest / Module / Effect / Combat / UI Runtime 연결
→ 플레이
→ Portal / BossKilled / PlayerDead 이벤트에 따라 SceneFlow 또는 RunRestart 처리
```

## 4. 데이터 계층

| 데이터 계층 | 구현 형태 | 변경 가능 여부 | 저장 여부 | 예시 |
| --- | --- | --- | --- | --- |
| Definition | ScriptableObject Asset | 런타임 변경 금지 | 저장하지 않음 | ModuleDefinition, QuestDefinition |
| RuntimeState | 순수 C# 클래스/struct | 런타임 변경 가능 | 필요 시 저장 | RunModuleBuildState |
| SaveData | 직렬화 전용 DTO | 저장/로드 시 변경 | 파일 저장 | PermanentSaveData |
| ViewModel | UI 표시용 변환 데이터 | 화면 갱신 시 변경 | 저장하지 않음 | HudViewModel |
| SceneObject | MonoBehaviour / Prefab | 씬 안에서 변경 | 직접 저장하지 않음 | PlayerController |
| AssetReference | Addressables 참조 | 런타임 로드/해제 | address key만 저장 가능 | Boss Prefab Reference |
| Scene Host | Hierarchy 또는 Prefab에 사전 배치된 MonoBehaviour | 참조와 상태를 연결 | 직접 저장하지 않음 | CombatSystemHost, QuestManagerHost |

### 4.1 Hierarchy 사전 배치 규칙

런타임 코드가 `GameObject` 또는 `Component`를 생성/부착하지 않는다. Bootstrap, Save, SceneFlow, Combat, Module, Quest, Reward는 씬 Hierarchy에 사전 배치된 Host Component가 소유한다. 순수 RuntimeState만 C# 객체로 생성한다.

## 9. Assembly Definition 기준

| asmdef | 포함 영역 | 참조 방향 |
| --- | --- | --- |
| Game.Core | Core, EventBus, Clock | 없음 |
| Game.Content | Definition, Registry | Core |
| Game.Save | SaveData, Migration | Core, Content |
| Game.SceneFlow | Loading, StageFlow | Core, Content, Save |
| Game.Gameplay | Player, Combat, Module, Boss, Quest | Core, Content, Save |
| Game.Presentation | UI, Camera, Audio, VFX | Core, Content, Gameplay |
| Game.Editor | Validator, ID Tool, Build Check | 모든 Runtime asmdef |
| Game.Tests | EditMode / PlayMode 테스트 | 테스트 대상 asmdef |

순환 참조는 금지한다. Presentation이 Gameplay를 표시할 수는 있지만, Gameplay가 특정 UI 클래스를 직접 참조하지 않는다.

| --- | --- | --- | --- |
| 게임 시작/종료 | GameStateMachine | GameRuntimeState | GameModeChanged |

| 저장/로드 | SaveSystem | SaveData | SaveLoaded, SaveWritten |

## 12. 최종 기준

기능은 시스템 소유권, 데이터는 Definition/Runtime/Save 경계, 파일은 `_Project` 중심 구조로 관리한다.

이 구조 없이 개별 시스템부터 구현하면 초반에는 빠르지만, 보스/모듈/퀘스트가 늘어날수록 콘텐츠 추가 비용과 버그가 급격히 증가한다.
