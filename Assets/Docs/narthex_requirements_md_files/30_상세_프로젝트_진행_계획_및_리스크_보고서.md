# 30. 상세 프로젝트 진행 계획 및 리스크 보고서

작성일: 2026-06-26

이 문서는 `Assets/Docs/narthex_requirements_md_files` 안의 요구사항 명세를 기준으로, 나르텍스 MVP를 실제 Unity 프로젝트에서 어떤 순서로 진행할지 단계별로 정리한 실행 계획이다.

목표는 단순한 기능 목록이 아니라, 이 계획대로 작업했을 때 생길 수 있는 문제점, 개선점, 좋은 점까지 미리 파악해 개발 중 판단 기준으로 쓰는 것이다.

---

## 1. 최종 결론

현재 문서 구조는 개발 착수 가능한 수준이다. 다만 실제 Unity 프로젝트는 아직 초기 생성 상태에 가깝기 때문에, 콘텐츠 제작을 바로 시작하면 위험하다.

따라서 첫 작업은 캐릭터, 보스, 스테이지 제작이 아니라 다음 기반을 고정하는 것이다.

```txt
패키지 / URP / 폴더 / asmdef / Boot / Registry / Save / SceneFlow / Validator / 최소 테스트
```

이 기반이 통과된 뒤에야 플레이어 조작, 전투, 모듈, 퀘스트, 스테이지, 보스 콘텐츠로 넘어간다.

## 4. 절대 진행 원칙

1. 콘텐츠보다 기반을 먼저 만든다.
2. Definition은 읽기 전용이다.
3. 런타임 변화는 RuntimeState에만 저장한다.
4. 저장 데이터는 stableId와 단순 DTO만 사용한다.
5. Manager 안에서 `questId`, `bossId`, `moduleId`별 `if/switch` 분기를 만들지 않는다.
6. 무거운 리소스는 Addressables 또는 명시적 로딩 소유권으로 관리한다.
7. Gameplay 판정은 Physics2D로 고정한다.
8. 전투 좌표계는 X/Y만 사용하고 Z는 렌더 정렬 전용으로 둔다.
9. UI는 uGUI + TextMeshPro 기준으로 진행한다.
10. Validator 실패 상태에서는 콘텐츠 제작과 빌드를 진행하지 않는다.

| M0-04 | 폴더 구조 생성 | `Assets/_Project`, `ThirdParty`, `AddressableAssetsData` | 23번 문서 구조와 일치 |

### 예상 문제점

| 문제 | 원인 | 대응 |
| --- | --- | --- |
| 패키지 설치 후 API 버전 차이 | Unity 6000.3과 패키지 최신 버전 차이 | 설치 직후 compile gate 실행 |

### 좋은 점

- 프로젝트가 초기 상태라 구조를 깨끗하게 고정하기 쉽다.
- Unity 버전은 문서 기준과 맞는다.

| Boss Gate | PatternPreview, ArenaBounds, BossKilled 보상 연결 |
| Loop Gate | NewGame to Death to Restart to BossKill to Result |
| Release Gate | 전시용 빌드에서 주요 루프 정상 |

## 7. 이 계획대로 진행했을 때의 문제점

| 문제점 | 설명 | 심각도 | 완화책 |

| Validator 개발 비용 | 처음에는 게임이 아니라 도구를 만드는 느낌이 강하다 | 중 | 필수 Validator부터 만들고 나머지는 Gate별 확장 |
| Addressables 학습/설정 부담 | 씬/프리팹/오디오 관리가 복잡하다 | 중 | AddressableLoadScope를 초기에 작게 구현 |
| 최종 보스 미정 | 후반 완성도와 전시 임팩트가 늦게 결정된다 | 상 | M11 placeholder로 전체 루프를 먼저 닫음 |
