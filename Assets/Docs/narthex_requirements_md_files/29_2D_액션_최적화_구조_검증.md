# 29. 2D 액션 최적화 구조 검증

## 1. 목적

이 문서는 현재 구조가 만들고자 하는 게임인 2D 액션 플랫포머 / 로그라이크 회차 진행에 최적인지 추가 검증하고, 실제 Unity 6.3 URP 개발 중 반복적으로 문제가 되는 제작 규약을 고정한다.

기존 22~28번 문서는 데이터 주도 구조, 저장, Addressables, Validator, 로드맵을 닫았다. 다만 2D 액션 게임은 데이터 구조만으로 충분하지 않다. 렌더러 선택, 스테이지 충돌 제작 방식, 카메라 룸, 이동 감각, 히트박스 좌표계, 보스 아레나 규약이 고정되어야 구현자가 같은 기준으로 계속 콘텐츠를 추가할 수 있다.

## 2. 공격적 검증 결론

| 항목 | 기존 상태 | 구조적 위험 | 보강 기준 |
| --- | --- | --- | --- |
| URP 렌더러 | URP로만 정의 | 2D Renderer와 Universal Renderer 혼용 시 조명, 머티리얼, 카메라 결과가 달라질 수 있음 | 기본은 URP 2D Renderer로 고정 |
| 스테이지 제작 | StageDefinition 존재 | 타일맵 충돌, 수작업 Collider, 포탈, 카메라 경계가 사람마다 다르게 만들어질 수 있음 | StageDefinition + RoomDefinition + StageRoot 규약 추가 |
| 카메라 | Cinemachine 사용 가능 | 룸 전환, 보스방 고정, 포탈 이동 때 카메라 경계 누락 가능 | CameraBounds와 CameraProfile을 데이터로 연결 |
| 이동 감각 | 이동 기능 목록 존재 | 점프 버퍼, 코요테 타임, 지면 판정이 없으면 2D 액션 조작감이 불안정 | PlayerMotorDefinition에 감각 수치 고정 |
| 전투 판정 | HitboxDefinition 존재 | 좌표 기준, 앵커, active frame, hitstop 규약이 없으면 모듈/보스 패턴 제작이 흔들림 | HitboxDefinition 좌표/시간/앵커 규약 고정 |
| 보스 아레나 | BossDefinition 존재 | 보스가 씬과 직접 결합되거나 패턴 테스트가 실제 스테이지에서만 가능해짐 | RoomDefinition이 보스 아레나를 소유하고 PatternPreview 테스트 추가 |
| 모듈/데우스 | Ability 구조 존재 | 데우스 연출 위치와 실제 판정 위치가 어긋날 수 있음 | ModuleCastAnchorDefinition으로 발동 원점 통일 |
| 밸런스 | 수치 데이터 분산 가능 | HP, 피해량, 모듈 포인트 보상이 여러 Definition에 흩어져 조정 비용 증가 | BalanceProfileDefinition과 플레이테스트 지표 추가 |

보강 후 판정은 다음과 같다.

```txt
데이터 주도 구조: 적합
Unity 6.3 URP 구현 안정성: 적합
2D 액션 플랫포머 제작 규약: 본 문서 기준으로 적합
즉시 개발 착수 가능 여부: 가능
```

## 3. URP 렌더러 결정

튜토리얼과 CHAPTER_01의 기본 렌더러는 URP 2D Renderer로 고정한다.

| 항목 | 기준 |
| --- | --- |
| 기본 Renderer | URP 2D Renderer |
| 기본 카메라 | Orthographic Camera + Cinemachine |
| 기본 시각 요소 | SpriteRenderer, TilemapRenderer, 2D Light, 2D Shadow |
| Gameplay 충돌 | Renderer와 무관하게 Physics2D |
| Universal Renderer | 실제 3D Mesh 연출이 핵심인 별도 씬에서만 RendererProfileDefinition으로 명시 |
| Custom Render Feature | 화면 연출 전용. 전투 판정, 무적, 피격, 퀘스트 조건에 영향 금지 |

Universal Renderer를 기본으로 쓰면 2D Light, Sprite, Tilemap 중심 제작에서 불필요한 렌더링 선택지가 늘어난다. 이 프로젝트는 2D 액션 플랫포머이므로 기본은 2D Renderer가 가장 안전하다.

### 3.1 CameraRenderProfileDefinition

```txt
CameraRenderProfileDefinition
→ profileId
→ rendererType: URP2D | Universal
→ usePixelPerfect
→ referenceResolution
→ assetsPPU
→ orthographicSize
→ allowPostProcess
→ volumeProfileReference
```

Pixel Perfect는 아트 방향이 픽셀아트로 확정된 경우에만 활성화한다. 픽셀아트가 아니라면 `usePixelPerfect = false`로 두고, Orthographic Size와 PPU 기준만 맞춘다.

## 4. 스테이지 제작 구조

StageDefinition은 씬 주소만 갖는 데이터가 아니라 스테이지 제작 규약의 원장이 되어야 한다.

```txt
StageDefinition
→ stageId
→ chapterId
→ sceneAddress
→ stageType: Tutorial | Normal | Boss | Result
→ defaultSpawnPointId
→ defaultCameraProfileId
→ roomIds
→ portalIds
→ encounterIds
→ preloadLabels
→ balanceProfileId
```

```txt
RoomDefinition
→ roomId
→ roomType: Tutorial | Traverse | Combat | BossArena
→ cameraBoundsId
→ entryGateIds
→ exitPortalIds
→ enemySpawnerIds
→ bossId
→ lockPolicy
→ clearConditionIds
→ musicProfileId
```

### 4.1 StageRoot 필수 오브젝트

```txt
StageRoot
├─ StageGeometryRoot
├─ VisualTilemapRoot
├─ CollisionTilemapRoot
├─ CameraBoundsRoot
├─ SpawnPointRoot
├─ EncounterRoot
├─ PortalRoot
├─ BossArenaRoot
└─ StageDebugGizmos
```

StageRoot는 StageDefinition을 참조한다. 개별 Enemy, Boss, Quest, Portal이 서로를 직접 찾지 않는다. 스테이지 진입 시 StageRuntime이 StageDefinition을 읽고 Room, Spawn, Portal, Encounter를 연결한다.

### 4.2 충돌 제작 규칙

| 제작 방식 | 사용 기준 | 필수 컴포넌트 |
| --- | --- | --- |
| 타일맵 기반 | 대부분의 지형 | TilemapCollider2D + CompositeCollider2D + Static Rigidbody2D |
| 수작업 Collider | 보스방, 특수 발판, 곡선 지형 | 전용 StageGeometry Collider2D |
| 일방향 발판 | 위에서만 착지하는 플랫폼 | StageOneWayPlatform Layer + PlatformEffector2D |
| Hazard | 피해 지형 | Hazard Layer + Trigger Collider2D + EffectDefinition |

Visual Tilemap과 Collision Tilemap은 분리한다. 보이는 타일을 바꿨다고 충돌이 자동으로 바뀌는 구조는 금지한다.

경사면은 CHAPTER_01 초기 범위에서는 사용하지 않는다. 나중에 추가할 경우 `PlayerMotorDefinition.maxSlopeAngle`과 GroundProbe 테스트를 먼저 추가한다.

## 5. 이동 감각 구조

2D 액션은 입력과 물리 처리 타이밍이 섞이면 조작감이 깨진다. 입력은 `Update`에서 수집하고, 실제 이동은 `FixedUpdate`에서 MovementMotor가 처리한다.

```txt
InputSystem
→ MovementIntent 생성
→ PlayerMovementController가 Intent 보관
→ FixedUpdate에서 PlayerMotor가 Rigidbody2D velocity 갱신
→ GroundProbe가 지면 상태 갱신
```

### 5.1 PlayerMotorDefinition

```txt
PlayerMotorDefinition
→ maxRunSpeed
→ groundAcceleration
→ groundDeceleration
→ airAcceleration
→ airDeceleration
→ jumpVelocity
→ variableJumpCutMultiplier
→ coyoteTime
→ jumpBufferTime
→ gravityScaleUp
→ gravityScaleDown
→ maxFallSpeed
→ dashDistance
→ dashDuration
→ dashCooldown
→ dashInvulnerabilityDuration
```

초기 추천값은 고정값이 아니라 시작점이다.

| 항목 | 초기 범위 |
| --- | --- |
| coyoteTime | 0.08~0.12초 |
| jumpBufferTime | 0.08~0.12초 |
| dashInvulnerabilityDuration | dashDuration 이하 |
| variableJumpCutMultiplier | 0.4~0.7 |

대시는 `Transform.position` 직접 이동으로 처리하지 않는다. MovementMotor 또는 Ability MoveAction이 소유한다. 블링크처럼 순간이동이 필요한 패턴만 TeleportAction을 사용하고, 목적지는 StageGeometry와 BossArenaBounds 안에서 안전 위치 검사를 통과해야 한다.

## 6. 전투 좌표계와 히트박스 규약

전투 좌표계는 다음으로 고정한다.

| 축 | 의미 |
| --- | --- |
| X | 좌우 이동과 공격 방향 |
| Y | 점프, 낙하, 높이 |
| Z | 렌더링 정렬 전용. Gameplay 판정에 사용 금지 |

모든 전투 개체는 다음 구조를 가진다.

```txt
CombatEntityRoot
├─ BodyCollider2D
├─ HurtboxRoot
├─ HitboxSpawnRoot
├─ AnchorSet
└─ FacingController
```

### 6.1 HitboxDefinition

```txt
HitboxDefinition
→ hitboxId
→ shape: Box | Circle | Capsule
→ anchorId
→ localOffset
→ size
→ facingMirror
→ activeDelay
→ activeDuration
→ targetLayerMask
→ maxHitsPerCast
→ hitInterval
→ effectIds
→ hitstopDuration
→ knockbackVector
→ knockbackSpace: Local | World | AwayFromSource
→ friendlyFirePolicy
```

Hitbox는 짧은 수명의 Trigger Runtime Object이다. 피해량, 넉백, 상태이상은 Hitbox가 직접 계산하지 않고 CombatSystem이 `DamagePacket`과 EffectDefinition으로 처리한다.

### 6.2 Active Frame 규약

판정 생성 타이밍은 ActionDefinition 타임라인이 소유한다.

```txt
AbilityDefinition
→ ActionDefinition
→ WaitAction
→ SpawnHitboxAction
→ MoveAction
→ SpawnProjectileAction
→ PlayVfxAction
→ PlaySfxAction
```

Animator Event는 VFX/SFX 보조 신호로만 사용한다. 실제 판정, 피해, 무적, 퀘스트 조건은 Animator Event에 의존하지 않는다.

### 6.3 Hitstop 기준

Hitstop은 선택 기능이 아니라 전투 감각을 위한 공통 기능으로 둔다.

| 상황 | 초기 범위 |
| --- | --- |
| 일반 공격 명중 | 0.03~0.06초 |
| 무거운 공격 명중 | 0.06~0.10초 |
| 보스 패턴 강한 피격 | 0.08~0.12초 |

Hitstop 중 UI, 로딩, 저장은 Unscaled Time으로 유지한다. 보스 패턴 타이머는 Gameplay Clock 기준이므로 hitstop의 영향을 받는다.

## 7. 보스 / Encounter 제작 구조

BossDefinition은 보스의 데이터 원장이고, BossArena 배치는 RoomDefinition이 소유한다.

```txt
RoomDefinition(BossArena)
→ bossId
→ cameraBoundsId
→ entryGateIds
→ exitPortalIds
→ lockPolicy: LockUntilBossKilled
→ clearConditionIds
→ musicProfileId
```

보스가 Scene, Portal, Camera를 직접 호출하지 않는다. BossKilled 이벤트가 발생하면 RoomRuntime이 잠금 해제, 포탈 활성화, 카메라 복귀를 처리한다.

### 7.1 PatternPreviewScene

보스 패턴은 실제 스테이지에 배치하기 전에 Preview Scene에서 단독 검증한다.

| 검증 | 기준 |
| --- | --- |
| PatternDefinition 로드 | 누락 참조 없음 |
| Ability 실행 | 예외 없음 |
| Hitbox Gizmo | active frame과 위치 확인 가능 |
| Arena Bounds | 패턴이 보스방 밖으로 이탈하지 않음 |
| Dummy Player | 피격, 회피, 넉백 확인 가능 |

헬테 블링크는 BossArenaBounds 안의 안전 후보 위치만 사용한다. 미노타우르스 돌진은 벽 충돌 Sensor를 필수로 두고, 플레이어 충돌 정지와 벽 충돌 4초 경직을 다른 결과로 분리한다.

## 8. 모듈 / 데우스 앵커 구조

데우스는 연출만 담당하는 것처럼 보이지만, 모듈 판정 원점이 흔들리면 전투 제작이 깨진다. 따라서 모듈의 실제 발동 위치도 데이터로 고정한다.

```txt
ModuleCastAnchorDefinition
→ anchorId
→ owner: Player | Deus | Boss | Enemy
→ anchorName
→ localOffset
→ facingMirror
→ targetPolicy: Forward | AimDirection | TargetPoint | Self
```

모듈 VFX와 Hitbox가 다른 위치에서 발생해야 할 경우에도 같은 AbilityDefinition 안에서 ActionDefinition을 분리한다. VFX는 DeusAnchor를 쓰고, 실제 판정은 PlayerForwardAnchor를 쓰는 식으로 명시한다.

## 9. 밸런스와 플레이테스트 데이터

수치가 각 Definition에 흩어지면 튜토리얼과 CHAPTER_01 난이도 곡선 검증이 어렵다. 초기 개발부터 BalanceProfileDefinition을 둔다.

```txt
BalanceProfileDefinition
→ balanceProfileId
→ playerBaseStats
→ enemyStatTable
→ bossStatTable
→ expRewardTable
→ modulePointRewardTable
→ moduleCostTable
→ stageDifficultyScalar
```

모든 수치를 BalanceProfile에만 몰아넣지는 않는다. Ability, Effect, BossPattern의 고유 수치는 해당 Definition에 둔다. 다만 CHAPTER_01 전체 곡선에 영향을 주는 HP, 피해량, 보상량, 비용은 BalanceProfile에서 한 번에 추적할 수 있어야 한다.

### 9.1 필수 플레이테스트 지표

| 지표 | 목적 |
| --- | --- |
| stageClearTime | 스테이지 길이와 난이도 확인 |
| damageTaken | 피격 압박 확인 |
| deaths | 회차 실패 지점 확인 |
| modulePointSpent | 모듈 포인트 경제 확인 |
| modulePickRate | 모듈 선호도 확인 |
| bossPatternHitRate | 보스 패턴 불합리성 확인 |
| retryCount | 로그라이크 반복 부담 확인 |

플레이테스트 지표는 저장 데이터와 분리한다. 개발용 TelemetryRuntimeLog 또는 DebugPanel에서 확인하고, 릴리스 저장에는 넣지 않는다.

## 10. 에셋 / 파일 구조 보강

기존 `_Project` 구조에 다음 하위 폴더를 추가한다.

```txt
Assets/_Project
├─ GameData
│  ├─ StageAuthoring
│  ├─ Camera
│  └─ Balance
├─ Prefabs
│  ├─ Stage
│  ├─ Characters
│  ├─ Bosses
│  └─ Gameplay
├─ Rendering
│  ├─ RendererAssets
│  │  ├─ URP2D
│  │  └─ Universal
│  └─ CameraProfiles
└─ Editor
   └─ Gizmos
```

가벼운 Definition은 ContentRegistry에서 직접 참조해도 된다. Scene, Boss Prefab, Enemy Prefab, VFX, Audio, 대형 SpriteAtlas는 Addressables로 관리한다.

## 11. 추가 Validator

| Validator | 실패 시 |
| --- | --- |
| RendererProfileValidator | 기본 씬이 URP 2D Renderer를 사용하지 않으면 빌드 차단 |
| StageAuthoringValidator | StageRoot 필수 오브젝트, StageDefinition 연결 누락 시 빌드 차단 |
| TilemapCollisionValidator | Visual Tilemap과 Collision Tilemap 혼용 시 빌드 차단 |
| CameraBoundsValidator | RoomDefinition의 cameraBoundsId 누락 시 빌드 차단 |
| MovementFeelConfigValidator | PlayerMotorDefinition 필수 수치 누락 시 빌드 차단 |
| HitboxCoordinateValidator | Z축 판정 사용, anchor 누락, activeDuration 0 이하 시 빌드 차단 |
| BossArenaValidator | BossArena의 lockPolicy, bossId, exitPortal 누락 시 빌드 차단 |
| PatternPreviewValidator | 보스 패턴 Preview 실행 실패 시 빌드 차단 |
| BalanceProfileValidator | StageDefinition의 balanceProfileId 누락 시 빌드 차단 |

## 13. 최종 판정

기존 구조는 데이터 주도 게임 시스템으로는 적합했다. 그러나 2D 액션 플랫포머의 제작 최적성까지 따지면 본 문서의 기준이 추가되어야 한다.

본 문서가 연결된 뒤의 최종 구조는 다음 조건을 만족한다.

- 개발자가 Unity 6.3 URP 프로젝트를 생성한 직후 렌더러 선택에서 흔들리지 않는다.
- 스테이지, 룸, 포탈, 카메라, 보스 아레나가 같은 데이터 흐름으로 연결된다.
- 이동 감각과 전투 판정이 콘텐츠마다 다르게 구현되지 않는다.
- 보스 패턴은 실제 보스방 투입 전 Preview Scene에서 검증할 수 있다.
- 모듈, 효과, 보스 패턴, 퀘스트, 밸런스가 서로 재사용 가능한 Definition 구조로 유지된다.

따라서 이 보강까지 반영한 현재 문서 구조는, 튜토리얼과 CHAPTER_01 개발을 지금 시작해도 되는 구조로 판정한다.
